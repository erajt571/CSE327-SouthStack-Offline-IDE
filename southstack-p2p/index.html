    Can’t connect / connection refused — fixes
    1. Wrong port (very common): Default server is :8000. If the phone URL shows :8001 (or anything else) and nothing listens there → refused. Use http://YOUR_IP:8000 or run PORT=8001 python3 serve_with_signal.py to match.
    2. Typo: Use 192.168.x.x, not 191.168. Click Apply to invite link & QR after fixing.
    3. Server on the host PC: In Terminal: cd southstack-p2p then python3 serve_with_signal.py — leave it running. No server → refused.
    4. Firewall: Allow incoming connections for Python (or port 8000) on the host.
    5. 127.0.0.1 on a phone means the phone itself — use the PC’s LAN link or QR after fixing the IP.
    6. Invite works in one browser but not another: Use the same URL and port as the host. Append ?nosw=1 to skip the service worker if an old cache blocks /api/southstack/. Tap Join room again after the host has started the room.
    7. Joined but “Devices in this room” shows only you / no shared job: WebRTC did not open between browsers. Add ?offline=1 to the URL on host and guest (same Wi‑Fi). Wait ~15s after join for ICE. Host: try New guest and a fresh invite.

    QR still points at localhost. Set LAN URL for phones above → Apply → use Start session again or Create room only + refresh QR.

    AI in this project

    Engine: WebLLM / MLC loads quantized models in the browser (Chrome + WebGPU recommended).

    WhatsApp-style Ask: one chat thread; any linked device can Ask; everyone sees the same bubbles; only one generation at a time. The coordinator runs WebLLM — chosen among devices that report WebGPU, then lowest peer id (so a guest phone usually won’t steal the role from a PC).

    If one machine sleeps or closes: with exactly two devices, the WebRTC session ends — you must reconnect (same room code, host uses New guest / guest taps Join room). This is not WhatsApp’s cloud sync; there is no central server holding your session. With three or more linked peers, others may stay connected and elect a new coordinator.

    Low-spec PC + phone: you cannot merge RAM/CPU across browsers. Use Start shared job to delegate subtasks to peers with WebGPU; for Ask, the coordinator device runs the full model.

    Solo: Ask runs entirely on this device. First visit downloads JS + weights; cache + IndexedDB help later.

    Checkpoints in IndexedDB are per browser profile. Reconnect the room after a full disconnect.
    Quick start (server)

    cd southstack-p2p then python3 serve_with_signal.py — open http://localhost:PORT (see terminal), Start session — show link & QR, same Wi‑Fi on other devices. Fallback: python3 -m http.server 8000 and manual SDP (Advanced below).

    Phone / tablet
    1. Same Wi‑Fi. Open http://<PC_LAN_IP>:PORT — not localhost.
    2. Ask while linked: coordinator runs the model. Solo on phone needs WebGPU (often missing).
    3. Invite / QR avoids typing codes. serve_with_signal.py auto-swaps SDP when possible.
    4. Use the Ask button on mobile; await promptCoding("…") is for desktop DevTools.
    Single PC
    1. Chrome + WebGPU.
    2. Wait for AI ready in the status bar.
    3. Ask or Console: await promptCoding("…").
    4. Two tabs: Connect devices + BroadcastChannel can auto-fill SDP.

    2. Same URL for everyone on Wi‑Fi.
    3. Host: create room → invite. Guest: join or open link.
    4. Manual SDP: host copy offer → guest paste → guest copy answer → host apply.
    5. Another device: host New guest per round.

    URL flags
    • ?offline=1 — no Google STUN (same LAN).
    • ?nosw=1 — skip service worker if cache breaks /api/southstack/.
    • ?easy=1 — host auto-start session + QR.
    • Invites: room=…, join=1, invite=1.
    Troubleshooting
    • Connection refused: wrong port, server off, or firewall.
    • 192.168.x.x not 191.168; set LAN URL + Apply before QR.
    • Only 1 device: WebRTC failed — ?offline=1, New guest.
    • WebGPU: recent Chrome; many phones can’t run solo.

    First visit needs network for WebLLM + weights; then cache + IndexedDB help.
